# 工业级能力基线

## 目标

平台不仅要“能生成向量”，还要具备可运维、可治理、可审计、可扩展的生产级能力。以下能力建议作为默认基线，而不是后补项。

## 可用性基线

- 控制面服务月可用性目标：`>= 99.9%`
- 数据面核心链路月可用性目标：`>= 99.9%`
- 关键服务部署方式：多副本、跨可用区、无单点
- 依赖治理：超时、重试、熔断、限流、隔离舱

## 性能基线

- 文本同步Embedding接口 P95：`< 50ms`
- 检索接口 P95：`< 100ms`
- 支持动态批处理、批量任务、GPU池化、热模型常驻
- 每个服务必须定义容量模型：QPS、并发数、显存、吞吐极限

## 数据可靠性基线

- 任务状态可恢复，异步任务幂等
- 向量写入支持去重、重试、补偿
- 索引构建支持快照、回滚、版本化
- 元数据和审计数据必须可追溯

## 安全与合规基线

- 全链路TLS
- 存储加密
- API Key 或 OAuth2/JWT 鉴权
- RBAC/ABAC 权限控制
- 审计日志默认开启且不可绕过
- 敏感数据支持私有化部署与脱敏处理

## 运维基线

- 指标、日志、链路追踪三件套必须打通
- 每个服务都有SLO、告警、Runbook
- 支持灰度发布、健康检查、回滚
- 关键任务支持人工重放与死信处理

## 成本治理基线

- 每个请求记录模型、租户、成本标签
- 区分缓存命中成本与实际推理成本
- 第三方API、本地GPU、存储三类成本单独核算
- 支持预算告警、租户配额与模型路由策略

## 工程基线

- 契约、配置、代码、部署分层管理
- 共享库最小化，避免隐式耦合
- 强制代码规范、静态检查、单测、集成测试
- 所有外部接口必须版本化

## 发布基线

- 主干开发，短分支合并
- CI执行lint、unit test、contract test
- CD支持dev/stage/prod多环境
- 生产发布使用灰度或分批滚动
