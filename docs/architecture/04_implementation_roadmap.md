# 平台实施路线图

## 1. 目标

本路线图用于将Embedding平台从架构蓝图推进到工业级落地，按照“先闭环、再生产、后平台化、再优化扩展”的原则分阶段实施。

路线图设计目标：

- 避免一次性建设过多能力导致周期过长和风险过高
- 保证每个阶段都有可交付成果和可验证业务价值
- 让架构、工程、运维、安全、成本治理同步演进，而不是后补

## 2. 总体实施策略

整体建议分为5个阶段：

1. Phase 0：方案定版与基础设施准备
2. Phase 1：MVP闭环
3. Phase 2：生产可用
4. Phase 3：平台化建设
5. Phase 4：多模态与高级优化

推进原则：

- 第一优先级是跑通最小可用链路
- 第二优先级是提升稳定性与生产可运维性
- 第三优先级是做平台治理与规模化复用
- 第四优先级是做多模态和性能成本的极致优化

## 3. 阶段拆分

### 3.1 Phase 0：方案定版与基础设施准备

#### 阶段目标

为后续开发建立统一技术标准、研发规范与基础运行底座，解决“怎么建、按什么规范建、部署到哪里”的问题。

#### 核心功能

- 完成平台总体方案、边界定义和服务拆分
- 定义API契约、事件契约、错误码规范
- 定义多租户模型、模型元数据模型、索引元数据模型
- 确定基础技术选型：网关、缓存、消息队列、元数据库、向量库、模型推理框架
- 初始化monorepo结构、CI流程和基础开发规范
- 准备dev/stage/prod环境基线

#### 重点服务范围

- `gateway`
- `tenant-auth`
- `model-registry`
- `task-orchestrator`
- `vector-store-proxy`
- `embedding-runtime`

当前阶段以接口和骨架为主，不做完整业务实现。

#### 关键依赖

- Kubernetes集群或等价运行环境
- 镜像仓库与CI/CD流水线
- PostgreSQL、Redis、Kafka、向量库实例
- 对接至少一个第三方模型与一个本地模型方案

#### 阶段产出

- 评审版总体架构文档
- 仓库骨架与目录规范
- OpenAPI/gRPC/事件契约初稿
- 开发环境和部署环境说明
- 安全与运维基线

#### 验收标准

- 团队对服务边界、目录规范、接口风格达成一致
- 基础依赖环境可创建、可访问、可用于后续联调
- 仓库可支撑后续服务并行开发

### 3.2 Phase 1：MVP闭环

#### 阶段目标

优先支持文本Embedding和基础检索，跑通“输入 -> 预处理 -> Embedding -> 向量入库 -> 检索返回”的最小闭环，让业务方尽快开始试用。

#### 核心功能

- 统一文本Embedding接口
- 支持单条同步Embedding
- 支持批量异步Embedding任务
- 支持文本切块、去重、基础清洗
- 支持结果缓存
- 支持向量写入和topK检索
- 提供基础监控、日志和追踪

#### 重点服务范围

- `gateway`
- `embedding-runtime`
- `task-orchestrator`
- `preprocess`
- `vector-store-proxy`
- `retrieval`
- `cache`

#### 关键依赖

- Phase 0 的接口与基础设施定版
- 至少一种文本向量模型稳定可用
- 至少一种向量库稳定可用

#### 阶段产出

- `POST /v1/embeddings`
- `POST /v1/tasks/embedding`
- `GET /v1/tasks/{taskId}`
- `POST /v1/retrieval/search`
- 基础数据入库任务链路
- 文本知识库小规模入库演示

#### 验收标准

- 单条文本Embedding链路可用
- 小规模批量任务可稳定完成
- 检索接口可返回topK结果
- 可观察到延迟、错误率、吞吐等基础指标

### 3.3 Phase 2：生产可用

#### 阶段目标

让平台具备正式承接业务流量的能力，从“能跑”提升到“稳定、可控、可恢复、可治理”。

#### 核心功能

- 租户、应用、凭证、权限模型落地
- 限流、熔断、超时、重试、降级策略落地
- 任务幂等、死信、补偿、人工重放
- 成本统计、配额控制、预算告警
- 索引生命周期管理与版本回滚
- 审计日志与敏感操作审计
- 多副本部署、健康检查、滚动发布

#### 重点服务范围

- `tenant-auth`
- `gateway`
- `cost-quota`
- `task-orchestrator`
- `index-manager`
- `vector-store-proxy`
- `embedding-runtime`

#### 关键依赖

- Phase 1 闭环稳定
- 业务试点场景明确SLA
- 运维与安全策略有明确接入要求

#### 阶段产出

- 多租户鉴权与授权体系
- 成本与用量统计面板
- 异常任务恢复和死信处理机制
- 生产环境部署清单
- 告警和Runbook

#### 验收标准

- 平台可用性达到既定目标
- 关键故障可定位、可恢复、可回滚
- 租户间数据和权限隔离生效
- 可看到按租户、按模型、按服务的成本数据

### 3.4 Phase 3：平台化建设

#### 阶段目标

从一个“Embedding服务集合”演进成统一平台底座，使新业务、新模型和新索引类型能标准化接入和运营。

#### 核心功能

- 模型注册中心与模型路由策略
- 模型灰度、热更新、回滚
- 索引管理控制能力
- SDK、CLI、标准接入模板
- 多环境配置治理
- 质量评估、A/B实验和效果对比
- 平台控制台基础能力

#### 重点服务范围

- `model-registry`
- `index-manager`
- `gateway`
- `tenant-auth`
- `retrieval`
- `packages/sdk`
- `packages/proto`

#### 关键依赖

- Phase 2 完成可用性和治理基础
- 模型接入频率增加，业务方复用诉求增强

#### 阶段产出

- 模型纳管与路由配置能力
- 平台统一SDK
- 新业务接入手册
- 实验评测基线
- 管理侧接口或控制台初版

#### 验收标准

- 新模型可按标准流程接入并灰度发布
- 新业务接入成本显著降低
- 不同模型和索引方案可做统一评估与治理

### 3.5 Phase 4：多模态与高级优化

#### 阶段目标

提升平台能力上限，支持图像、音频等多模态场景，并持续优化吞吐、延迟与成本。

#### 核心功能

- 图像Embedding
- 音频Embedding或音频转文本后Embedding
- 混合检索和重排服务
- 动态维度适配
- GPU池化与智能调度
- 向量压缩、冷热分层、索引压缩
- 漂移监控、质量评估、容量预测

#### 重点服务范围

- `embedding-runtime`
- `preprocess`
- `retrieval`
- `vector-store-proxy`
- `model-registry`

#### 关键依赖

- 平台已有稳定控制面和数据面能力
- 新模态业务需求明确
- GPU资源与成本模型可支撑扩展

#### 阶段产出

- 多模态Embedding能力
- 混合检索与重排能力
- 成本和性能调优体系
- 高阶质量评估与容量管理机制

#### 验收标准

- 支持至少两种模态
- 高峰吞吐和单位成本较Phase 2显著优化
- 多模态能力可在至少一个业务场景中落地

## 4. 服务实施优先级

### P0：首批必须建设

- `gateway`
- `embedding-runtime`
- `task-orchestrator`
- `vector-store-proxy`
- `retrieval`
- `cache`

这些服务直接决定平台是否能形成最小闭环。

### P1：生产化必须补齐

- `tenant-auth`
- `cost-quota`
- `index-manager`
- `preprocess`

这些服务决定平台是否能正式承载生产流量。

### P2：平台化与规模化能力

- `model-registry`
- `packages/sdk`
- 平台控制台
- 实验评测与质量治理模块

这些服务决定平台是否能真正降低接入成本并支持长期运营。

## 5. 关键依赖关系

### 5.1 控制链路依赖

- 没有`gateway`，外部无法统一接入
- 没有`tenant-auth`，生产场景无法实现租户隔离和权限控制
- 没有`model-registry`，多模型管理和灰度发布会失控

### 5.2 数据链路依赖

- 没有`preprocess`，长文本和脏数据会严重影响效果和成本
- 没有`embedding-runtime`，平台无核心产出能力
- 没有`vector-store-proxy`，底层向量后端切换和治理成本过高
- 没有`retrieval`，平台无法支撑检索类业务闭环

### 5.3 治理链路依赖

- 没有`cost-quota`，成本不可见、不可控
- 没有`index-manager`，索引重建、回滚和压缩会缺少统一控制
- 没有运行手册与告警，平台无法工业级运维

## 8. 推荐执行顺序

推荐按以下顺序推进核心服务实现：

1. `gateway`
2. `embedding-runtime`
3. `vector-store-proxy`
4. `retrieval`
5. `task-orchestrator`
6. `preprocess`
7. `tenant-auth`
8. `cost-quota`
9. `index-manager`
10. `model-registry`

原因如下：

- 先把同步链路和检索闭环跑通
- 再补离线链路和预处理
- 再补生产治理能力
- 最后补平台运营和模型治理能力

## 9. 风险提示

### 风险1：阶段1做得过重

如果在MVP阶段提前引入过多治理和平台能力，会拖慢最小闭环交付。

建议：

- MVP只做最小必需能力
- 管理控制台、复杂调度、评测系统放到后续阶段

### 风险2：阶段2才开始考虑安全和运维

如果安全、审计、监控完全后置，后续补齐代价会很高。

建议：

- 在Phase 0就定义规范
- 在Phase 1就预埋日志、指标、追踪字段

### 风险3：共享库失控

平台型仓库容易把业务逻辑堆到`common`中，导致耦合失控。

建议：

- `common`只放稳定基础能力
- 业务逻辑保留在服务内部

## 10. 总结

实施上建议遵循以下节奏：

- 第一阶段先拿到可运行闭环
- 第二阶段把闭环打磨成生产系统
- 第三阶段把生产系统打磨成平台产品
- 第四阶段再追求多模态与极致优化

这样可以确保平台每个阶段都有明确价值产出，也能更稳地向工业级能力演进。
